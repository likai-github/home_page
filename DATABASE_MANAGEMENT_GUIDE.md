# 🗄️ 数据库管理功能使用指南

## 概述

新增的数据库管理功能提供了一个可视化界面，让你可以轻松查看和管理数据库状态，无需手动访问 API 端点或使用 D1 Studio。

**位置：** 管理后台 → 数据库

---

## 🎯 主要功能

### 1. 数据库状态概览 📊

实时显示数据库的关键信息：

- **当前版本** - 数据库当前的迁移版本
- **最新版本** - 系统支持的最新版本
- **数据库状态** - 是否需要更新
- **表数量** - 当前数据库中的表数量

**操作按钮：**
- 🔄 **刷新状态** - 重新获取最新状态
- ⬆️ **执行更新** - 当有新版本时显示，点击执行数据库迁移

---

### 2. 数据表信息 📋

显示所有数据库表的列表和统计信息：

**表卡片显示：**
- 表名
- 记录数量
- 表描述

**操作：**
- 点击表卡片查看详细信息
- 🔄 **刷新** - 重新加载表列表

**表详情对话框包含：**
- **基本信息** - 表名、记录数
- **字段结构** - 所有字段的详细信息（名称、类型、是否非空、默认值、是否主键）
- **示例数据** - 前 5 条记录的 JSON 格式数据

---

### 3. 迁移历史 📜

显示所有已执行的数据库迁移记录：

**每条记录显示：**
- 版本号（v1, v2, v3...）
- 迁移名称
- 执行时间
- 状态（✅ 已完成）

**操作：**
- 🔄 **刷新** - 重新加载迁移历史

---

### 4. 危险操作 ⚠️

提供数据库重置功能（需谨慎使用）：

**重置数据库：**
- 删除所有表和数据
- 需要输入 `RESET` 确认
- 不可恢复

---

## 📖 使用场景

### 场景 1：检查数据库状态

**步骤：**
1. 进入管理后台 → 数据库
2. 查看"数据库状态"卡片
3. 确认当前版本和表数量

**预期结果：**
- 当前版本：v10
- 表数量：14
- 状态：最新

---

### 场景 2：执行数据库更新

**何时需要：**
- 部署新代码后
- 数据库状态显示"需要更新"
- 添加了新的迁移脚本

**步骤：**
1. 进入管理后台 → 数据库
2. 查看状态，如果显示"需要更新"
3. 点击 **⬆️ 执行更新** 按钮
4. 等待更新完成
5. 查看成功消息

**预期结果：**
```
✓ 数据库迁移完成 (v9 → v10)
```

---

### 场景 3：查看表结构

**步骤：**
1. 进入管理后台 → 数据库
2. 在"数据表信息"区域找到目标表
3. 点击表卡片
4. 查看字段结构和示例数据

**用途：**
- 了解表结构
- 调试数据问题
- 验证数据是否正确

---

### 场景 4：验证迁移执行

**步骤：**
1. 执行数据库更新后
2. 查看"迁移历史"区域
3. 确认新的迁移记录已添加

**预期结果：**
- 看到 v1 到 v10 的所有迁移记录
- 每条记录都有执行时间

---

### 场景 5：重置数据库（开发/测试）

**⚠️ 警告：此操作会删除所有数据！**

**何时使用：**
- 开发环境测试
- 数据库结构损坏
- 需要从头开始

**步骤：**
1. 进入管理后台 → 数据库
2. 滚动到"危险操作"区域
3. 点击 **🗑️ 重置数据库**
4. 阅读警告信息
5. 输入 `RESET` 确认
6. 点击"确认重置"
7. 等待完成后，点击 **⬆️ 执行更新** 重新创建表

## 🔗 API 端点

数据库管理功能使用以下 API 端点：

```
GET  /api/database/status                  - 获取数据库状态
GET  /api/database/tables                  - 获取所有表信息
GET  /api/database/tables/:name/columns    - 获取表字段
GET  /api/database/tables/:name/data       - 获取表示例数据
GET  /api/database/migrations              - 获取迁移历史
POST /api/database/migrate                 - 执行迁移
POST /api/database/reset                   - 重置数据库
```

## 💡 最佳实践

### 1. 定期检查状态

建议每次部署后检查数据库状态：
- 确认版本是否最新
- 验证表数量是否正确
- 查看迁移历史

### 2. 更新前备份（生产环境）

虽然迁移是增量的，但在生产环境执行更新前：
- 在 D1 Studio 中导出数据
- 记录当前版本号
- 准备回滚方案

### 3. 使用表详情调试

遇到数据问题时：
- 查看表结构确认字段
- 查看示例数据验证内容
- 对比预期结构

### 4. 谨慎使用重置

重置数据库会删除所有数据：
- ✅ 开发环境：可以随意使用
- ⚠️ 测试环境：确认无重要数据
- ❌ 生产环境：绝对不要使用

## ❓ 常见问题

### Q1: 数据库状态显示"需要更新"怎么办？

A: 点击"执行更新"按钮，系统会自动执行所有待执行的迁移。

### Q2: 表数量不对怎么办？

A: 
1. 检查数据库状态，确认版本
2. 如果版本不是最新，执行更新
3. 如果版本已是最新但表数量不对，可能需要重置数据库

### Q3: 如何查看某个表的具体数据？

A: 点击表卡片，在弹出的对话框中查看"示例数据"部分。

### Q4: 迁移失败怎么办？

A: 
1. 查看错误消息
2. 检查 Cloudflare Pages 日志
3. 如果是结构冲突，可能需要重置数据库

### Q5: 重置后需要做什么？

A: 重置后需要：
1. 点击"执行更新"重新创建所有表
2. 重新注册第一个用户（会自动成为超级管理员）
3. 重新配置 API 平台

## 🎯 快速操作指南

### 首次部署后

```
1. 登录管理后台
2. 进入"数据库"
3. 点击"执行更新"
4. 等待完成
5. 验证表数量为 14
```

### 代码更新后

```
1. Git push 部署
2. 等待构建完成
3. 登录管理后台
4. 进入"数据库"
5. 如果显示"需要更新"，点击"执行更新"
```

### 遇到问题时

```
1. 进入"数据库"
2. 查看当前状态
3. 查看迁移历史
4. 点击表卡片查看详情
5. 根据情况决定是否需要重置
```
